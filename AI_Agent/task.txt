# Motivation and Long-Term Architecture Vision

The current RTS agent uses a Large Language Model (LLM) to directly interpret raw game state and generate commands for Command & Conquer: Generals Zero Hour. While this already demonstrates promising behavior, pure LLM control has several major limitations in real-time strategy environments:

* excessive context size
* unstable decision making
* weak long-term planning
* inconsistent micro control
* poor economic optimization
* repeated low-value actions
* limited mechanical precision
* high inference cost and latency

RTS games are extremely demanding AI environments because they require both:

1. high-level strategic reasoning
2. fast low-level mechanical execution

LLMs are strong at:

* strategic adaptation
* semantic reasoning
* flexible planning
* interpreting complex situations

However, LLMs are weak at:

* frame-by-frame precision
* repetitive optimization
* tactical consistency
* long-horizon economic efficiency
* precise micro-management

The goal of the next development stages is therefore to evolve the system from a “pure prompt-driven agent” into a hybrid RTS architecture where:

* the engine handles structured game analysis,
* learned policies handle fast execution,
* and the LLM focuses mainly on strategic reasoning and adaptation.

This architecture is closer to how advanced game AI systems are designed in research and production environments.

The development roadmap is divided into five major steps:

1. Improve engine preprocessing
2. Reduce prompt/context size massively
3. Add replay-derived strategic features
4. Add imitation-learning policy
5. Use the LLM mainly for planning and adaptation

These steps progressively move intelligence out of the raw prompt and into specialized systems that are:

* faster,
* more stable,
* more scalable,
* and more strategically competent.

The final objective is not merely to “make the LLM smarter,” but to create a layered RTS AI architecture where each component performs the tasks it is best suited for.

Step 1 — Improve Engine Preprocessing:
Goal:
Transform raw game state into high-value structured RTS information before the LLM sees it.

The engine should do the “RTS bookkeeping,” not the LLM.

1.1 Build a Game Analyzer Layer

Raw state:

{
  "money": 5000,
  "units": [...]
}

Better processed state:

{
  "economy": {
    "income_level": "low",
    "idle_dozers": 1,
    "supply_count": 1
  },
  "military": {
    "army_strength": 0.42,
    "anti_air": false
  }
}

You want semantic RTS features.

1.2 Detect Strategic Situations

Add engine-side detectors:

Examples
{
  "enemy_rush_detected": true,
  "enemy_air_threat": false,
  "floating_resources": true,
  "base_under_attack": false,
  "can_expand_safely": true
}

This massively improves decisions.

1.3 Add Unit Role Classification

Instead of:

{
  "name": "AmericaTankCrusader"
}

Add:

{
  "role": "frontline_tank",
  "threat_type": "ground"
}

This reduces reasoning burden.

1.4 Add Economy Metrics

Very important.

Compute:

{
  "income_per_minute": 3200,
  "resource_collection_efficiency": 0.71,
  "idle_worker_count": 2,
  "production_idle_time": 14
}

LLMs are terrible at inferring these from raw state.

1.5 Add Combat Evaluations

Compute approximate battle outcomes.

Example:

{
  "local_battles": [
    {
      "location": [1200, 800],
      "friendly_strength": 0.65,
      "enemy_strength": 0.41,
      "win_probability": 0.78
    }
  ]
}

This becomes extremely useful.

1.6 Add Expansion Analysis

Example:

{
  "expansions": [
    {
      "location": [1350, 2200],
      "safe": true,
      "distance_score": 0.62,
      "resource_value": 0.94
    }
  ]
}
1.7 Add Tactical Zones

Partition map into regions:

{
  "map_control": {
    "north": "enemy",
    "center": "contested",
    "south": "friendly"
  }
}

This helps strategic planning enormously.
